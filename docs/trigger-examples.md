# Trigger Examples

## Purpose

这一页用来解决协议型 Skill 最常见的理解门槛：

- 什么请求会触发 `ai-native-loop`
- 什么请求不该触发它
- 它触发后会把请求改写成什么样
- 大概会落在轻 / 中 / 强哪一档介入

## Rule Of Thumb

当问题的核心不只是“要一个答案”，而是：

- 输入太乱
- 执行在漂
- 反馈没被吸收
- 下一轮不知道怎么开始

这时就应该优先考虑 `ai-native-loop`。

如果问题已经非常清晰、范围很窄、只差一个直接答案，通常不需要触发它。

## Trigger Decision

按这个顺序判断是否触发：

1. 先判断：用户要的是不是一个直接答案？
2. 如果不是，再判断：当前卡点是不是闭环问题，而不是知识缺口问题？
3. 再数信号：下面 6 条至少命中 2 条才优先触发

信号列表：

- 需求、材料、历史尝试或报错混在一起
- 当前轮目标、成功信号或下一检查点不清楚
- 已有输出，但不知道保留什么、丢弃什么、如何进入下一轮
- 反馈存在，但没有归因结构
- 多角色或多 Agent 协作边界不清
- 同类问题已重复至少两轮

出现以下任一条，通常不要触发：

- 单点事实查询
- 翻译、改格式、改单个小项
- 目标和验收标准都清楚，只差直接执行
- 更窄的领域 skill 已经足够

如果用户明确说“用 ai-native-loop 帮我诊断 / 整理 / 重写下一轮输入”，则应优先触发，不再要求凑满 2 条信号。

## Should Trigger

### Example 1: 编码任务上下文很乱

用户请求：

> 我在改一个复杂功能，需求、代码状态、报错和之前试过的方法全混在一起了。帮我整理一下怎么继续。

为什么应触发：

- 当前问题不是只缺“答案”
- 而是输入材料混乱、执行路径不稳定

推荐介入：`medium`

触发后更可能的改写方向：

- 重建任务包
- 分离当前状态、约束、报错与历史尝试
- 定义下一检查点和验证方式

### Example 2: 研究做了一轮但不知道怎么吸收反馈

用户请求：

> 我已经做了一轮研究，也拿到一些 AI 输出，但我不知道哪些反馈真的值得进入下一轮。

为什么应触发：

- 这是典型的“反馈没有进入再输入”问题

推荐介入：`medium`

触发后更可能的改写方向：

- 区分正确性、完整性、方向性和杠杆性反馈
- 输出一张反馈归因卡
- 形成下一轮更好的研究输入

### Example 3: 多 Agent 协作失衡

用户请求：

> 我现在让几个 agent 分别做不同部分，但他们返回的东西很难汇总，感觉一直在重复劳动。

为什么应触发：

- 问题核心是协作协议与交接结构失衡

推荐介入：`strong`

触发后更可能的改写方向：

- 明确共享目标
- 重写子任务边界与交接物
- 建立统一的回收格式

### Example 4: 写作过程中输出能看但不好用

用户请求：

> AI 已经帮我写了一版，但结构和重点都不太对，我不知道应该直接改这版，还是重写下一轮输入。

为什么应触发：

- 这是典型“已有输出无法直接转化为下一轮推进”的问题

推荐介入：`light` 到 `medium`

触发后更可能的改写方向：

- 识别哪些内容保留
- 哪些应该丢弃
- 生成更适合下一轮的输入包

## Usually Should Not Trigger

### Example 5: 单一事实查询

用户请求：

> 帮我查一下 React 19 的 `use` 是做什么的。

为什么通常不触发：

- 这是单点知识查询
- 问题核心不是闭环质量，而是事实获取

更合适的处理：

- 直接回答或使用更窄的资料检索能力

### Example 6: 已经非常明确的机械执行

用户请求：

> 把这个标题翻译成英文。

为什么通常不触发：

- 任务极窄
- 没有上下文重组、反馈吸收或再输入问题

### Example 7: 只差一步的小修小补

用户请求：

> 这段 CSS 帮我把按钮颜色改成蓝色。

为什么通常不触发：

- 如果上下文成熟且动作简单，直接改即可
- 除非用户真正卡的是需求表达与协作链路

## Borderline Cases

### Example 8: 用户说“帮我想一想”

这类请求要先判断：

- 他是缺一个点子
- 还是缺一个工作闭环

如果更像是：

- 不知道怎么推进
- 不知道这轮要什么
- 不知道怎么把已有输出接到下一轮

那就应该触发。

如果只是要一组想法，不一定需要触发。

### Example 9: 用户想让 AI 先帮自己整理一下

这通常是 `ai-native-loop` 的高价值入口。

因为“整理一下”背后常常意味着：

- 输入结构不稳定
- 任务包不成形
- 下一步不清楚

## What It Usually Outputs

触发后，`ai-native-loop` 更倾向于输出这些东西，而不是泛建议：

- 一个 AI-ready task packet
- 一份人机 / 多 Agent 分工安排
- 一张反馈归因卡
- 一份下一轮输入包
- 一个更合适的检查点

## Quick Heuristic

可以用一句话判断：

> 如果你真正卡住的不是“答案是什么”，而是“怎么把这件事和 AI 协作得更顺”，就该触发 `ai-native-loop`。

再压缩成给 Agent 的一句：

> 只有当阻塞点落在输入、执行、反馈或再输入之间的连接处时，才触发 `ai-native-loop`。

如果要验证触发边界是否真的稳定，而不是只看例子，请继续使用 [trigger-regression-suite.md](trigger-regression-suite.md)。
