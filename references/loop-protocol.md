# Loop Protocol

## 核心主张

这个 skill 处理的不是单个请求，而是一个持续往返的知识工作闭环：

1. 输入重写
2. 执行分工
3. 反馈识别
4. 再输入改写

它的目标不是把一轮做满，而是把下一轮做得更好。

## 一轮工作的最小任务包

把任何原始请求压缩为以下字段：

- `objective`
  当前这一轮真正要推进什么。
- `current_state`
  现在已经有什么，缺什么，卡在哪里。
- `artifacts`
  代码、文档、报错、材料、已有输出、外部约束。
- `constraints`
  时间、质量、格式、风险、不可做事项。
- `success_signal`
  什么结果算这轮有效，不要求完美，但必须可判断。
- `next_checkpoint`
  下一次停下来检查什么。

如果连这 6 项都说不清，说明任务还不适合直接求最终答案。

## 阶段 1：输入重写

目标：把“原始表达”变成“ai 可执行表达”。

重写时优先处理：

- 模糊目标，改成明确推进对象。
- 混杂上下文，拆成当前状态与补充材料。
- 一次说太多，收敛到本轮最关键推进面。
- 只有动作没有标准，补上成功信号。

典型输出：

- 精炼任务包
- 需要补齐的 1 到 3 个关键缺口
- 更适合 agent 执行的输入形式

## 阶段 2：执行分工

目标：确定人、ai、agent 各自承担什么。

默认分工原则：

- 人负责目标判断、权衡、验收和不可逆决策。
- ai 负责草拟、展开、压缩、比较、提案、重组。
- agent 负责可执行、可验证、可回收的步骤。

执行分工时至少回答：

- 哪一步是探索，哪一步是生产。
- 哪些输出需要人工确认。
- 哪些结果必须沉淀为下一轮可复用材料。

## 阶段 3：反馈识别

目标：读取信号，而不是只看“像不像答案”。

优先辨认：

- 结果对不对。
- 结果全不全。
- 过程是不是浪费。
- 分工是不是错位。
- 方向是不是跑偏。

反馈必须进入可归因结构，否则不会稳定转化为下一轮改进。

## 阶段 4：再输入改写

目标：把本轮经验折叠成下一轮更好的起点。

再输入至少包含：

- 本轮保留什么。
- 本轮丢弃什么。
- 下一轮要补什么上下文。
- 下一轮要改变什么提问或执行方式。

优先做最小有效改动，不要每轮都重建全部协议。

## 何时说明闭环已断

满足任一情况，可判为循环断裂，需要升级介入：

- 目标与交付物无法对齐。
- 输出无法被现有反馈评估。
- 多次失败却没有归因变化。
- 用户和 ai 在重复同一种低质量交互。

## 本文件的使用方式

- 任务定义模糊时，先读这一份。
- 需要把单次任务提升为长期工作协议时，反复回到这一份。
- 如果已经明确要用轻、中、强哪一层，再跳到 `intervention-matrix.md`。
