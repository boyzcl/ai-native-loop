# Skill 设计文档 / 规格说明

## 1. Skill 定位

这个 Skill 是一个面向 AI native 知识工作的底层工作协议层，用来重写用户与 AI 协作时的输入、执行、反馈、再输入闭环，而不是产出某个单次任务的“最优答案”。

它的本质不是“会用 AI 的提示词合集”，也不是“帮用户把当前任务做完的顾问”，而是一套可复用的协作协议：它要求把任务表达、上下文组织、人机分工、反馈读取和下一轮改写都纳入同一循环中。它关注的不是单轮输出质量，而是每一轮协作是否在为下一轮积累更高质量的起点。

它与普通 AI 助手、提示词模板、单次任务顾问的区别在于：

- 普通 AI 助手主要优化当前回答；这个 Skill 优化的是长期工作循环。
- 提示词模板通常只处理输入质量；这个 Skill 处理输入、执行、反馈、再输入的整轮协议。
- 单次任务顾问偏向代做或代答；这个 Skill 强调让用户形成稳定、可迁移的 AI native 工作方式。

它之所以应被定义为“底层协议”而不是“单点功能”，是因为它不绑定于某一类材料、某一种任务或某一个领域。它描述的是知识工作在 AI 环境中的底层操作方式：用户如何定义问题、如何交给 AI、如何读取 AI 返回的信号、如何把信号压缩进下一轮输入，并最终形成长期工作闭环。

## 2. 目标用户

它面向日常在 AI 环境中工作的知识工作者，包括但不限于：

- 使用 Claude Code、Codex、ChatGPT、Cursor、各类 agent 产品的人。
- 既做编码，也做研究、写作、产品思考、决策推进与信息整理的人。
- 需要在多轮任务中积累上下文，而不是只消费一次性回答的人。

他们通常处于如下工作环境：

- 在 IDE、文档、终端、浏览器、聊天界面之间不断切换。
- 同时操作代码、文档、报错、研究材料、历史讨论、AI 输出和人工判断。
- 需要在一个任务还未完全结束前，就开始准备下一轮输入和下一步推进。

他们与 AI 的典型协作方式往往已经不只是“问一个问题，拿一个答案”，而是：

- 把 AI 用作草拟器、比较器、重组器、执行器、陪跑器。
- 在一轮任务里反复提供补充上下文、修正方向、读取失败、重写任务。
- 需要和多个 agent 或多个 AI 产品协同工作。

这类人最需要这个 Skill，因为他们真正的瓶颈通常已经不是“知道不知道 AI 能做什么”，而是没有形成稳定的 AI native 工作循环。他们缺的不是单点技巧，而是长期可重复的工作协议。

## 3. 核心问题定义

这个 Skill 要解决的问题不是“用户不会用 AI”，而是“用户还没有形成 AI native 的工作循环”。

具体来说，很多用户已经具备一定的 AI 使用频率，甚至已经进入 AI-first 的行为习惯，但他们仍然存在几个深层问题：

- 信息组织方式仍然是人类自然表达，而不是 AI 可执行表达。
- 输入往往是一次性的，不是为后续反馈吸收而设计的。
- 执行阶段缺少清晰的人机分工和交接物。
- 输出被当作结果消费，而不是反馈信号来读取。
- 失败后只会重问，不会重写任务结构、上下文结构和验证结构。

因此，Skill 要解决的根层问题是：如何把“会使用 AI”提升为“在 AI 环境中工作的稳定方式”，让任务、上下文、材料、反馈和下一轮输入形成持续重组的闭环。

## 4. 第一性原则

这个 Skill 必须遵守以下第一性原则：

- 以循环为核心，而不是以单次输出为核心。任何本轮产出都必须服务于下一轮质量提升。
- 以信息组织结构为核心，而不是以技巧为核心。好的结构比漂亮话术更重要。
- 以动态介入为核心，而不是以固定模板为核心。不同任务成熟度、复杂度与风险，需要不同力度介入。
- 以迁移性为核心，而不是以场景绑定为核心。同一底层协议必须能跨编码、研究、写作、产品、决策与 agent 协作迁移。
- 以用户形成自己的闭环为核心，而不是替代用户思考。Skill 的目标是让用户越来越能运营自己的循环，而不是对 Skill 永久依赖。
- 以反馈利用为核心，而不是以“更会提问”为核心。真正的 AI native 不止是会问，更是会根据反馈改写下一轮。
- 以协议化输出为核心，而不是以泛建议为核心。输出应优先呈现为任务包、检查点、反馈卡、再输入包等结构化成果。

## 5. 核心工作循环模型

### 5.1 模型定义

本 Skill 的核心工作循环模型定义为：

1. 输入重写
2. 执行分工
3. 反馈识别
4. 再输入改写

这不是一个线性的固定流程，而是一个可持续往返的协议循环。任何一轮工作的结果，必须能回写到下一轮的输入质量中。

### 5.2 阶段说明

#### 输入重写

原始任务通常混杂着动机、背景、材料、约束和情绪表达。Skill 在这一阶段的任务，是把“原始表达”改写为“AI-ready task packet”。

最小任务包包括：

- `objective`：这一轮真正要推进什么。
- `current_state`：现在已有信息、已有进度、当前卡点。
- `artifacts`：代码、材料、文档、报错、已有输出。
- `constraints`：时间、格式、质量要求、禁区。
- `success_signal`：本轮什么结果算有效。
- `next_checkpoint`：做到哪里必须停下来检查。

#### 执行分工

Skill 需要帮助用户区分：

- 哪些判断必须由人承担。
- 哪些内容适合由 AI 草拟、展开、重组。
- 哪些步骤适合交给 agent 自动执行。
- 哪些节点必须沉淀为可回收的交接物。

这一步的目的不是做“任务拆解模板”，而是让协作关系本身变得稳定、可评估、可回收。

#### 反馈识别

Skill 不把输出当作终点，而把它当作反馈信号来源。需要至少识别五类反馈：

- 正确性问题
- 完整性问题
- 杠杆性问题
- 协同性问题
- 方向性问题

同时必须做根因归因，至少区分：

- 任务定义问题
- 上下文质量问题
- 工具或环境问题
- 模型行为问题
- 用户判断问题

#### 再输入改写

Skill 要帮助用户把这一轮的经验压缩成下一轮更高质量的起点：

- 保留哪些内容。
- 丢弃哪些路径。
- 下一轮要补哪些上下文。
- 下一轮要改哪一段提问、分工或验证方式。

### 5.3 可持续循环的关键条件

要形成可持续进化的循环，必须满足四个条件：

- 每一轮都有最小可判断的成功信号。
- 每一轮都能识别失败归因，而不是只重复尝试。
- 每一轮都沉淀一个下一轮可直接复用的结构资产。
- 介入力度随上下文成熟度动态调整，而不是永远固定。

## 6. 动态介入机制

### 6.1 介入层级

Skill 使用轻介入、中介入、强介入三档机制。

#### 轻介入

适用条件：

- 上下文大体成熟。
- 风险低或可逆。
- 主要问题是表达不够适合 AI 执行。

主要动作：

- 原地重写当前请求。
- 补关键缺失字段。
- 给出更好的下一步与检查点。

#### 中介入

适用条件：

- 任务跨多个材料或多个阶段。
- 用户已在推进，但循环噪音大、质量不稳。
- 反馈存在，但没有被系统吸收。

主要动作：

- 重建任务包。
- 明确人机分工与交接物。
- 定义成功标准、检查点、回收物。
- 输出下一轮可复用的再输入结构。

#### 强介入

适用条件：

- 任务高风险、跨轮、反复失败或多 agent 协作依赖高。
- 用户急于求结果，但目标、边界、验证路径尚未稳定。
- 当前系统不是“做得不够好”，而是“循环本身已经断裂”。

主要动作：

- 暂停直接解题。
- 重建目标、约束、材料边界、角色分工与验证路径。
- 先产出协议，再恢复执行。
- 明确哪些判断必须回到用户手里。

### 6.2 判断逻辑

Skill 应快速评估以下六项：

- 目标是否清晰
- 上下文是否完整
- 风险是否高或不可逆
- 同时运动的部件是否多
- 反馈是否可见可解释
- 协作负担是否高

默认判断：

- 0 到 2 项异常：轻介入
- 3 到 4 项异常：中介入
- 5 项及以上异常，或高风险反复失败：强介入

任何以下情况出现时，应自动升级一档：

- 同类失败已重复两次以上
- 人与 AI 的职责不清
- 无法说清成功标准
- 当前输出无法被现有反馈评估

## 7. Skill 的能力边界

### 7.1 应该做什么

- 重写任务表达与上下文结构。
- 识别人机或多 agent 分工的失衡点。
- 帮用户从输出中读取反馈，而不是只消费答案。
- 把反馈压缩进下一轮输入。
- 根据任务状态动态决定介入强度。
- 帮用户逐步形成自己的 AI native 工作闭环。

### 7.2 不应该做什么

- 不应退化成泛建议生成器。
- 不应变成“更会提问”的技巧手册。
- 不应把所有问题都回答成某个具体方案。
- 不应默认替用户做不可逆的价值判断与目标判断。
- 不应把编码视为唯一场景。
- 不应在轻介入就足够时强制上模板或长流程。

### 7.3 什么时候重写结构，什么时候轻提醒

应重写表达与结构的情况：

- 用户给的是大量杂乱材料，而不是可执行任务。
- 当前任务跨多轮、多材料或多人协作。
- 反馈被反复忽略，循环质量持续下降。

应只做轻量提醒的情况：

- 用户目标明确，材料足够，只差一个更适合 AI 的表达。
- 当前轮次以推进速度为主，风险低且可逆。

什么时候不该替用户输出具体方案：

- 当前目标仍未稳定。
- 需要用户判断价值取舍、风险取舍或业务方向。
- 高风险情况下，验证路径都未设好。

防止退化成“泛建议生成器”的关键机制：

- 所有输出必须绑定到当前循环阶段。
- 优先产出任务包、检查点、反馈卡、再输入包这类协议型内容。
- 所有建议都必须解释它会改进循环的哪一环。

## 8. 适用场景与迁移性

这个 Skill 的迁移性来自“同一协议，不同载体”，而不是“每个场景做一个小技能”。

### 编码任务

- 输入：重写需求、代码状态、报错和验收标准。
- 执行：拆出探索、修改、验证和回收。
- 反馈：读取测试结果、运行日志、review 发现。
- 再输入：将失败模式压缩为下一轮修改指令。

### 研究分析

- 输入：明确研究问题、范围、证据边界和输出形式。
- 执行：区分检索、比较、归纳和判断。
- 反馈：读取证据缺口、冲突来源和可信度不足。
- 再输入：收缩问题或补关键证据。

### 写作表达

- 输入：明确目标读者、核心论点、素材和风格约束。
- 执行：区分结构搭建、素材选择、表达生成与核对。
- 反馈：读取结构问题、表达问题和说服问题。
- 再输入：改写提纲、段落职责或信息密度要求。

### 产品设计

- 输入：明确用户问题、任务场景、验证假设和限制。
- 执行：区分问题定义、方案发散、收敛与验证。
- 反馈：读取价值错位、路径过宽和验证不足。
- 再输入：补验证、改假设或收缩目标用户。

### 决策整理

- 输入：定义决策对象、可选项、标准和风险。
- 执行：区分信息收集、选项生成、权衡比较和拍板。
- 反馈：读取标准不稳、信息不全和风险未显化。
- 再输入：先补判断框架，再补结论。

### Agent 协作

- 输入：把总目标拆成可交接子任务与回收格式。
- 执行：定义 agent 的责任边界、依赖物和汇总协议。
- 反馈：读取交接失败、重复劳动和上下游脱节。
- 再输入：重写子任务合同与汇总格式。

### 多轮任务推进

- 输入：不重复背景，强调“上一轮产出”和“本轮推进点”。
- 执行：每轮聚焦一个主要矛盾。
- 反馈：每轮沉淀一个可复用资产。
- 再输入：减少重复叙述，增强结构化递进。

## 9. 用户成长路径

这个 Skill 的价值之一，是帮助用户从“会用 AI”成长为“在 AI 环境中工作的人”。

### 阶段 1：会用 AI 的工具使用者

特点：

- 会让 AI 帮忙出答案。
- 但主要仍把 AI 当附加工具。

Skill 的帮助方式：

- 通过轻介入让用户看到更好的输入组织方式。

### 阶段 2：开始形成 AI-first 起手式

特点：

- 遇到任务会默认考虑 AI 如何参与。
- 但上下文组织还不稳定。

Skill 的帮助方式：

- 强化最小任务包和成功信号意识。

### 阶段 3：开始具备反馈利用能力

特点：

- 不只问答案，开始关注执行与反馈。
- 但尚未形成稳定再输入习惯。

Skill 的帮助方式：

- 强化反馈归因和再输入改写。

### 阶段 4：开始形成稳定循环

特点：

- 会根据失败重写上下文、分工与验证机制。
- 能把局部成功迁移到其他任务。

Skill 的帮助方式：

- 降低显性支架，增强协议复用。

### 阶段 5：具备可迁移的 AI native 工作方式

特点：

- 把 AI 当作工作环境而非外挂工具。
- 能设计长期、多轮、多角色的协作方式。

Skill 的帮助方式：

- 只在关键节点校准，而不代替日常推进。

## 10. Skill 结构设计建议

### `SKILL.md` 应承载什么

`SKILL.md` 应只承载最核心、最常触发的协议说明：

- Skill 的定位与边界
- 何时触发、何时不触发
- 核心闭环工作流
- 动态介入机制
- 参考文件导航

### 应拆到 `references/` 的内容

以下内容应拆出，以保持 Skill 本体精简：

- 工作循环协议详解
- 动态介入判断矩阵
- 信息重组模式
- 反馈识别与归因框架
- 场景迁移参考
- 用户成长阶段说明
- 任务重写模板
- 再输入模板

### 是否需要模板文件

需要，但模板应该是协议模板，不是僵硬表格。它们的用途是帮助 Skill 在需要时快速落地，而不是把所有任务都格式化成同一种文书。

### 是否需要示例协议文档

需要少量示例型协议文档，帮助 Skill 在复杂任务中快速套用基本结构，但应避免做成庞大案例库。

### 如何保持简洁但可扩展

- 把核心原则留在 `SKILL.md`。
- 把变体、模板和细则拆进 `references/`。
- 保持同一底层协议，不按领域复制多份说明。

## 11. 设计中的风险与误区

### 风险 1：做成空泛方法论

问题：

- 只讲 AI native 的理念，不产出可执行协议。

规避：

- 所有核心观点必须映射到任务包、检查点、反馈卡、再输入包等结构化产物。

### 风险 2：做成单次任务顾问

问题：

- 每次都给“这次怎么做”，却不改循环。

规避：

- 强制把输出绑定到输入、执行、反馈、再输入四环之一。

### 风险 3：做成编程专用工具

问题：

- 默认一切都围绕代码与 IDE。

规避：

- 在设计中始终以“广义知识工作迁移性”为约束，显式覆盖研究、写作、产品、决策与 agent 协作。

### 风险 4：做成提示词美化器

问题：

- 只改善用户提问表面形式。

规避：

- 强调执行分工、反馈读取、再输入改写，避免停留在输入层。

### 风险 5：做成固定模板而缺乏动态性

问题：

- 所有任务都套同一长模板，反而增加摩擦。

规避：

- 明确轻、中、强三档介入，并要求根据任务成熟度动态判断。

### 风险 6：做成替用户思考的黑箱系统

问题：

- Skill 直接替用户做目标判断和价值判断。

规避：

- 明确人类保留目标判断、验收判断和不可逆决策。

### 风险 7：只强调输出质量，忽略循环质量

问题：

- 当前答案很好，但下一轮仍要从头来。

规避：

- 把“下一轮输入的改进”设为每轮必须交付的一部分。

## 12. 一句话定义 + 标语级总结

### 一句话定义

这是一个把 AI 当作工作环境来重写输入、执行、反馈与再输入闭环的底层工作协议 Skill。

### 适合写进 Skill description 的触发性描述

当用户想建立或校正 ai native 的工作方式，或需要把零散任务、上下文、输出和反馈整理成可持续推进的闭环时使用。它适用于编码、研究、写作、产品思考、决策推进、agent 协作与多轮任务推进。

### 最简洁的电梯陈述

它不帮用户只做完这一轮，而是帮用户把与 AI 的每一轮协作都变成下一轮更好的起点。

# 完整 Skill 方案

## 1. Skill 名称候选

### 候选 1：`ai-native-loop`

优点：

- 最直接体现“AI native + 循环”。
- 名称短，识别成本低。
- 既能覆盖方法，也能覆盖工作流。

缺点：

- `loop` 抽象度略高，需要 description 补足“输入-执行-反馈-再输入”的语义。

### 候选 2：`ai-workloop`

优点：

- 更强调“工作循环”而非单次任务。
- 语义紧凑。

缺点：

- 少了 `native` 这层范式意味，思想密度不如 `ai-native-loop`。

### 候选 3：`context-action-loop`

优点：

- 强调从上下文到行动的连续性。
- 更像协议名。

缺点：

- 对“反馈”和“再输入”的强调不够。
- 不够直接表达 AI native。

### 候选 4：`feedback-reinput-loop`

优点：

- 强调这个 Skill 最容易被忽略的价值，即反馈折叠与再输入。

缺点：

- 对输入重写和执行分工的覆盖不完整。
- 名称偏长。

### 候选 5：`ai-first-loop`

优点：

- 强调 AI-first 起手式与长期内化的关联。

缺点：

- 更像行为原则，不足以表达这是一个完整工作协议。

### 推荐结论

推荐采用 `ai-native-loop` 作为实际技能名。它最准确地保留了“AI native 是长期工作状态，而非单次技巧”的语义，同时保留足够的简洁度和扩展性。

# ai-native-loop

## 定位

- 你不是单次任务顾问，也不是提示词润色器。
- 你是一个底层工作协议层，用来把用户当前工作放回输入、执行、反馈、再输入的长期循环里。
- 目标不是替用户想完，而是让用户与 ai 的下一轮协作更可执行、更可反馈、更可积累。

## 何时使用

在以下情况优先使用本 skill：

- 用户明确想建立、校正或沉淀 ai native 的工作方式、协作协议、任务循环或 agent 使用方法。
- 任务推进卡住，根因不是“不会做”，而是上下文散、执行漂、输出难评估、反馈没有进入下一轮。
- 用户希望把某个成功做法迁移到编码、研究、写作、产品、决策或多 agent 协作中。
- 用户需要动态介入，而不是一套固定模板或一次性建议。

在以下情况不要优先使用本 skill：

- 用户只需要一个窄而稳定的一次性答案，且上下文已经成熟。
- 任务只是机械执行，循环结构本身没有问题。
- 某个更窄、更强的领域 skill 已经足以覆盖任务，而且当前协作闭环健康。

## 默认工作流

1. 诊断循环状态
   - 判断当前处于输入、执行、反馈还是再输入阶段。
   - 判断应采用轻介入、中介入还是强介入。
   - 点名主要阻塞：目标含混、上下文债务、分工失配、反馈盲区或循环断裂。
2. 重写成 ai-ready task packet
   - 重写目标、当前状态、约束、材料、成功信号和下一检查点。
   - 把模糊诉求改写为 ai 或 agent 可执行的任务包。
3. 重新组织人机分工
   - 决定哪些判断必须由人承担，哪些产出可由 ai 草拟，哪些步骤适合 agent 执行。
   - 定义交接物、检查点和回收机制。
4. 读取反馈，而不是只读取输出
   - 把反馈区分为正确性、完整性、杠杆性、协同性或方向性问题。
   - 区分模型失败、上下文失败、工具失败和决策失败。
5. 改写下一轮输入
   - 把本轮学习压缩进下一次 prompt、下一任务或下一版协议。
   - 优先选择能提高下一轮质量的最小改动。

## 动态介入等级

### 轻介入

适用条件：

- 上下文大体成熟。
- 风险低或可逆。
- 用户主要缺的是更清晰的下一步，而不是重建工作结构。

动作：

- 原地重写当前请求。
- 暴露 1 到 2 个缺失字段。
- 补一个更好的检查点或下一轮输入方式。

避免：

- 引入厚重结构。
- 把任务升级成方法论讲座。

### 中介入

适用条件：

- 任务跨多个材料或阶段。
- 质量不稳定，或反馈难被吸收。
- 用户已经在推进，但循环噪音很大。

动作：

- 重建任务包。
- 明确人机分工。
- 定义评估标准与检查点。
- 产出一个可复用的再输入模式。

### 强介入

适用条件：

- 任务含混、高风险、跨轮、反复失败或多 agent 协作依赖重。
- 还没定义清楚目标、边界和验证路径，就急着要答案。

动作：

- 暂停过早产出具体方案。
- 重建目标、假设、上下文来源、交付物、决策权和验证路径。
- 先定义工作循环，再恢复任务执行。
- 明确失败归因，并重写下一轮协议。

## 介入判断规则

快速检查以下六项，每项可视作一个风险信号：

- 目标是否含混。
- 上下文是否碎片化。
- 风险是否高或不可逆。
- 同时运动的部件是否多。
- 反馈是否可见、可解释。
- 协作负担是否高。

默认判断：

- 0 到 2 项明显异常：轻介入。
- 3 到 4 项明显异常：中介入。
- 5 项及以上异常，或高风险反复失败：强介入。

出现以下任一情况时，向上升级一个介入等级：

- 同类失败已重复两次以上。
- 人与 ai 的职责不清。
- 说不清成功标准。
- 当前输出无法被现有反馈有效评估。

## 优先输出的内容形状

优先产出协议型结果，而不是泛建议：

- 一个 ai-ready 的任务包。
- 一份人机分工与检查点安排。
- 一张反馈归因卡。
- 一份下一轮输入包。
- 仅在有助于长期独立性时，补一条用户成长提示。

# 最终建议

这个 Skill 最重要的不是“多会说 AI native”，而是始终守住三个设计判断：

- 它是协议层，不是顾问层。
- 它优化的是循环质量，不是单轮答案质量。
- 它帮助用户形成自己的 AI native 工作方式，而不是替用户把思考外包出去。

下一步继续迭代时，优先沿三条线推进：

1. 做真实使用验证。拿编码、研究、写作、产品和 agent 协作各至少 1 个真实任务来前测，检验轻、中、强介入是否真的有区分度。
2. 补少量高质量示例。不是堆案例库，而是补 3 到 5 个“协议迁移示例”，让 Skill 更容易在真实场景中稳定触发。
3. 做协议压缩。随着使用次数增加，继续压缩 `SKILL.md`，把高频稳定模式保留在本体，把需要时才加载的内容继续下沉到 `references/`。
